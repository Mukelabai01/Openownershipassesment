# Database Schema

## Overview

The schema consists of four main tables:

1. **auth_user** (Django built-in): User accounts
2. **applications_application**: Workflow applications
3. **audits_auditlog**: Immutable event log
4. **auth_group** (Django built-in): User roles (e.g., "reviewer")

## Detailed Schema

### auth_user

Django's built-in user model. Extended via User.objects.get(id=...).

| Column       | Type         | Constraints             | Notes                 |
| ------------ | ------------ | ----------------------- | --------------------- |
| id           | SERIAL       | PRIMARY KEY             | Auto-generated        |
| username     | VARCHAR(150) | UNIQUE, NOT NULL        | Username for login    |
| email        | VARCHAR(254) | Unique, blank           | Email address         |
| first_name   | VARCHAR(150) | Blank                   | User's first name     |
| last_name    | VARCHAR(150) | Blank                   | User's last name      |
| password     | VARCHAR(128) | NOT NULL                | Hashed (PBKDF2)       |
| is_staff     | BOOLEAN      | NOT NULL, DEFAULT False | Admin access          |
| is_superuser | BOOLEAN      | NOT NULL, DEFAULT False | Full Django admin     |
| is_active    | BOOLEAN      | NOT NULL, DEFAULT True  | Can login             |
| last_login   | TIMESTAMP    | Nullable                | Last successful login |
| date_joined  | TIMESTAMP    | DEFAULT now()           | Account creation time |

**Role Detection**:

- **Applicant**: Any authenticated user
- **Reviewer**: `user.is_staff == True` (checked in `WorkflowService.is_reviewer()`)

**Future**: Consider `auth_group` membership for more flexible RBAC.

### applications_application

Workflow applications submitted by users.

| Column       | Type         | Constraints               | Notes                      |
| ------------ | ------------ | ------------------------- | -------------------------- |
| id           | UUID         | PRIMARY KEY               | Generated by uuid4()       |
| owner_id     | INT          | FOREIGN KEY, NOT NULL     | → auth_user(id)            |
| title        | VARCHAR(255) | NOT NULL                  | Application title          |
| status       | VARCHAR(50)  | NOT NULL, DEFAULT 'DRAFT' | See Status Choices below   |
| content      | JSONB        | NOT NULL                  | Flexible data store        |
| created_at   | TIMESTAMP    | NOT NULL, DEFAULT now()   | Creation time              |
| updated_at   | TIMESTAMP    | NOT NULL, DEFAULT now()   | Last modification time     |
| submitted_at | TIMESTAMP    | Nullable                  | First SUBMITTED transition |

**Status Choices**:

```python
class Status(TextChoices):
    DRAFT = 'DRAFT', 'Draft'
    SUBMITTED = 'SUBMITTED', 'Submitted'
    UNDER_REVIEW = 'UNDER_REVIEW', 'Under Review'
    APPROVED = 'APPROVED', 'Approved'
    REJECTED = 'REJECTED', 'Rejected'
```

**Content Example**:

```json
{
  "description": "A machine learning pipeline for data processing...",
  "technologies": ["Python", "TensorFlow", "Kubernetes"],
  "team_size": 5,
  "budget_usd": 50000,
  "timeline_weeks": 12,
  "objectives": [
    "Build data pipeline",
    "Train ML model",
    "Deploy to production"
  ]
}
```

**Indexes**:

- `(owner_id)`: Fast lookups of user's applications
- `(status)`: Fast filtering by status
- `(created_at)`: Time-based sorting
- `(owner_id, status)`: Common filter combination

---

### audits_auditlog

Immutable audit trail of all state transitions.

| Column         | Type        | Constraints             | Notes                                  |
| -------------- | ----------- | ----------------------- | -------------------------------------- |
| id             | UUID        | PRIMARY KEY             | Generated by uuid4()                   |
| application_id | UUID        | FOREIGN KEY, NOT NULL   | → applications_application(id)         |
| actor_id       | INT         | FOREIGN KEY, Nullable   | → auth_user(id), NULL if system action |
| from_status    | VARCHAR(50) | NOT NULL                | Previous status                        |
| to_status      | VARCHAR(50) | NOT NULL                | New status                             |
| comment        | TEXT        | Nullable                | Transition comment/reason              |
| metadata       | JSONB       | NOT NULL, DEFAULT {}    | Additional context                     |
| created_at     | TIMESTAMP   | NOT NULL, DEFAULT now() | When transition happened               |

**Immutability**:

- No UPDATE or DELETE operations allowed (enforce at ORM level)
- Only INSERT (enforced by schema design)

**Metadata Example**:

```json
{
  "rejection_reason": "Budget exceeds limit",
  "reviewer_notes": "Needs cost reduction",
  "ip_address": "192.168.1.100"
}
```

**Indexes**:

- `(application_id, created_at)`: Fast audit trail retrieval per application

## Relationships

### User ← → Application (One-to-Many)

```sql
ALTER TABLE applications_application
ADD CONSTRAINT fk_owner
FOREIGN KEY (owner_id) REFERENCES auth_user(id) ON DELETE CASCADE;
```

**Cardinality**: A user can have many applications; each application has one owner.
**On Delete**: If user is deleted, cascade delete their applications (and audit logs).

### Application ← → AuditLog (One-to-Many)

```sql
ALTER TABLE audits_auditlog
ADD CONSTRAINT fk_application
FOREIGN KEY (application_id) REFERENCES applications_application(id) ON DELETE CASCADE;
```

**Cardinality**: An application has many audit events; each audit event belongs to one application.
**On Delete**: If application is deleted, cascade delete its audit logs.

### AuditLog → User (Many-to-One, Optional)

```sql
ALTER TABLE audits_auditlog
ADD CONSTRAINT fk_actor
FOREIGN KEY (actor_id) REFERENCES auth_user(id) ON DELETE SET NULL;
```

**Cardinality**: An audit event may or may not have an actor (NULL if system action).
**On Delete**: If user is deleted, set actor_id to NULL (preserve history).

## Migrations

### Initial Migration (0001_initial.py)

**applications app**:

- Create `Application` model with UUID primary key
- Create indexes on (owner_id), (status), (created_at)
- Set default status = 'DRAFT'

**audits app**:

- Create `AuditLog` model with UUID primary key
- Create index on (application_id, created_at)
- Set actor_id as nullable ForeignKey to auth.User with SET_NULL on delete

**Run migrations**:

```bash
python manage.py makemigrations
python manage.py migrate
```

## Query Patterns

### List User's Applications

```python
apps = Application.objects.filter(owner_id=user_id).order_by('-created_at')
```

### List Applications by Status

```python
submitted = Application.objects.filter(status=Application.Status.SUBMITTED)
```

### Get Application with Related Audit Log

```python
app = Application.objects.get(id=app_id)
audit_trail = AuditLog.objects.filter(application_id=app_id).order_by('created_at')
```

### Get Most Recent Transition

```python
latest = AuditLog.objects.filter(application_id=app_id).latest('created_at')
```

### Count Approvals by Reviewer

```python
from django.db.models import Count
approvals = AuditLog.objects \
    .filter(to_status=Application.Status.APPROVED) \
    .values('actor_id') \
    .annotate(count=Count('id')) \
    .order_by('-count')
```

## Performance Considerations

### Table Sizes

- **auth_user**: O(users) - typically ~100-1000 rows
- **applications_application**: O(applications) - grows with submissions
- **audits_auditlog**: O(applications × transitions) - typically 3-5× larger than applications

### Slow Queries to Avoid

❌ `SELECT * FROM audits_auditlog WHERE actor_id = ?` (missing index)
✅ `SELECT * FROM audits_auditlog WHERE application_id = ? ORDER BY created_at` (indexed)

### Optimization Tips

1. Use `select_related()` when fetching related users
2. Use `prefetch_related()` for reverse relations
3. Consider denormalizing frequently-joined data (e.g., application count in user profile)
4. Archive old audit logs to a separate table after 2 years

---

## Security Considerations

### Password Storage

- Django uses PBKDF2 by default (salted, iterated hashing)
- Never store plain-text passwords
- Use Django's `make_password()` when creating users

### Audit Log Integrity

- Audit logs are immutable (enforced at ORM level)
- Include actor_id to track who made changes
- For critical transitions, store metadata (IP, timestamp, approval count)

### PII in Audit Logs

- Consider logging only status transitions, not sensitive content
- If logging content changes, mask sensitive fields (e.g., "**_REDACTED_**")
- Implement access controls so only admins/auditors can view full logs
